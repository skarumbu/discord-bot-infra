# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Terraform infrastructure for the Discord bot. Manages all Azure resources — container registry, container app, and storage — for both dev and prod environments. The bot source lives in the separate [`discord-bot-app`](https://github.com/skarumbu/discord-bot-app) repo.

## Structure

```
modules/
  container_registry/   # ACR + AcrPull managed identity
  container_app/        # Log Analytics, Container App Environment, Container App
  storage_account/      # Storage account + karma table (Azure Table Storage)
environments/
  dev/                  # Dev environment config + tfvars
  prod/                 # Prod environment config + tfvars
```

Each environment calls all three modules. Modules are designed to be swappable — the `source` line is the only thing that would change if moving to AWS/GCP.

## Commands

```bash
# From environments/dev or environments/prod:
terraform init
terraform plan
terraform apply
terraform apply -var="image=discordbotprodacr.azurecr.io/discord-bot:sha-<short-sha>"
```

## Authentication

Uses OIDC (federated identity) — no static credentials. Requires `az login` locally or the GitHub Actions workflow in CI. ARM environment variables used in CI:

```
ARM_CLIENT_ID
ARM_TENANT_ID
ARM_SUBSCRIPTION_ID
ARM_USE_OIDC=true
```

## Secrets

Sensitive values are passed via environment variables, never committed to tfvars:

- `TF_VAR_subscription_id` — Azure subscription ID
- `TF_VAR_secret_env_vars` — JSON map of secret env vars injected into the container app (e.g. `{"DISCORD_TOKEN":"..."}`)

The `KARMA_STORAGE_CONNECTION_STRING` is **not** in `TF_VAR_secret_env_vars` — it is generated by Terraform from the storage account output and merged in automatically in `environments/*/main.tf`.

## CI/CD

`.github/workflows/terraform.yml`:
- **Push to `main`** → `terraform apply` in `environments/prod`
- **PR to `main`** → `terraform plan` in `environments/prod`

Before applying, the workflow reads the currently deployed container image from Azure (`az containerapp show`) and passes it as `-var="image=..."` to avoid resetting the image to the placeholder default.

The container image itself is managed by the `discord-bot-app` CI pipeline, which runs `terraform apply -var="image=<new-image>"` after each successful build.

## State

Remote state stored in Azure Blob Storage:
- Resource group: `tfstate-rg`
- Storage account: `discordbottfstate`
- Container: `tfstate`
- Keys: `discord-bot-infra/dev/terraform.tfstate` and `discord-bot-infra/prod/terraform.tfstate`

The tfstate storage account must exist before running `terraform init`. It is managed manually (not by this Terraform config).

## Gotchas

- **Storage account naming:** Azure storage account names must be lowercase alphanumeric, max 24 chars, globally unique. The `storage_account` module strips hyphens from `name_prefix` and appends `karma` (e.g. `discordbotprodkarma`).
- **Image variable:** `var.image` has a placeholder default. Always pass the real image tag explicitly when applying in prod — either via CI or `-var="image=..."`.
- **`terraform apply` from infra repo:** Without passing `-var="image=..."`, Terraform will reset the container to the placeholder. The CI workflow handles this automatically; for manual applies, fetch the current image first with `az containerapp show`.
